# 呈现-素材组织思路 (综合版)

> 本文档旨在指导如何从一个已完成的课设项目中，挖掘出具有**学术说服力**和**工程美感**的汇报素材，并构建一套完整的"代码库考古与叙事合成"（CANS）框架。
>
> **核心原则**：本文档提供的是**方法论框架**而非固定模板。Agent 应根据具体项目的特性，**因地制宜地发掘**最能体现该项目价值的指标和素材。

---

## 第一章 总体原则：素材组织的核心思想

### 1.1 动态分析理念

> **关键原则**：不同项目有不同的"亮点基因"，Agent 需要首先理解项目本身，再决定挖掘什么。

| 原则 | 说明 |
|------|------|
| **项目导向** | 先理解项目做什么、解决什么问题，再决定关注哪些指标 |
| **特性发掘** | 识别项目的独特之处，而非套用通用模板 |
| **价值驱动** | 挖掘能证明项目价值的素材，而非填充所有维度 |
| **因地制宜** | 根据项目类型、规模、目标灵活调整分析重点 |

### 1.2 核心理念

| 要点 | 说明 |
|------|------|
| **明确做了什么** | 知道我们在这个项目做了什么，为什么选择这个技术方案 |
| **理解物理意义** | 搞明白底层机制背后的原理 |
| **体现深度思考** | 体现我们对问题和方案的思考 |
| **量化指标** | 要有可量化的、**与项目目标相关**的指标 |
| **客观自评** | 知道系统边界和实现水平 |
| **坦诚问题** | 把遇到的问题也大方放上去 |
| **数据支撑** | 有合理的数据支持结论 |

### 1.3 叙述展开方式

- 用**学术规范的格式**讲一个完整的故事
- 遵循"**需求分析 → 架构设计 → 模块实现 → 系统集成 → 性能验证**"的叙述线

## 第二章 项目特性发掘：Agent 的首要任务

> **在挖掘素材之前，Agent 必须先理解项目**

### 2.1 项目特性识别 Prompt

```
请先回答以下问题，帮助我理解这个项目的核心特性：

1. **项目定位**：这个项目解决什么问题？目标用户/场景是什么？

2. **技术领域**：属于什么技术领域？

3. **核心挑战**：项目面临的主要技术挑战是什么？

4. **关键指标**：对于这个项目，什么指标最能证明它的成功？请列出 3-5 个**与项目目标直接相关**的核心指标。

5. **参照基准**：有什么"金标准"可以作为对比基准？（如：官方库、理论公式、行业标准）

6. **技术亮点**：项目中有哪些值得展示的技术决策或实现？

7. **权衡取舍**：项目中有哪些需要展示的 Trade-off 分析？
```

### 2.2 常见项目维度参考（仅作启发，非硬性要求）

以下是一些常见的分析维度，**Agent 应根据项目实际情况选择适用的维度**：

| 维度 | 可能关注的方向（示例） | 呈现目的 |
|------|------------------------|----------|
| **精度维** | 与金标准的误差、数值稳定性 | 证明算法实现的正确性 |
| **效率维** | 执行时间、复杂度、加速比、吞吐率 | 证明方案的工程优越性 |
| **收敛维** | 收敛曲线、稳态误差、自适应行为 | 证明闭环系统的自适应能力 |
| **资源维** | 硬件占用、功耗、时序裕量 | 证明资源利用率与鲁棒性 |
| **鲁棒维** | 边界条件、异常场景下的表现 | 证明系统的可靠性 |

**注意**：上表仅为启发，实际维度应由 Agent 根据项目分析得出。

## 第三章 素材挖掘：构建全维度的证据链

> **以下维度作为思考框架，根据项目特性选择性使用**

### 3.1 维度一：系统架构与设计（理论之美）

#### 挖掘方向

- **系统拓扑**：项目的整体结构是什么样的？
- **模块划分**：核心模块有哪些？它们如何协作？
- **数据流动**：数据如何在系统中流转？
- **设计决策**：为什么选择这种架构？有什么权衡考量？

#### Agent 自主发掘指令

```
请分析项目结构，回答：
1. 项目的核心模块有哪些？它们的职责是什么？
2. 模块之间的依赖关系如何？请生成模块依赖图
3. 数据在系统中如何流动？请描述关键数据通路
4. 架构设计中有哪些值得展示的决策或权衡？
```

---

### 3.2 维度二：核心实现与技术亮点（实现之美）

#### 挖掘方向

- **核心算法**：项目的核心逻辑是什么？
- **技术难点**：实现过程中的关键挑战？
- **优化手段**：采用了哪些优化技术？
- **创新点**：有哪些独特的实现方式？

#### Agent 自主发掘指令

```
请分析项目代码，识别：
1. 项目的核心算法/逻辑在哪里？请解释其工作原理
2. 代码中有哪些值得注意的技术实现？
3. 哪些代码段最能体现项目的技术深度？
4. 相比朴素实现，这些优化带来了什么改进？
```

---

### 3.3 维度三：性能验证与量化分析（量化之美）

#### 挖掘方向

- **关键指标**：哪些指标最能证明项目成功？
- **参照基准**：与什么进行对比？（金标准）
- **多维探索**：参数变化如何影响性能？
- **数据可视化**：如何呈现测试结果？

#### Agent 自主发掘指令

```
基于对项目的理解，请：
1. 确定最能体现项目价值的量化指标
2. 识别合适的参照基准（金标准）
3. 建议如何设计对比实验
4. 如有多个可调参数，建议进行敏感度分析
```

---

### 3.4 维度四：工程实践与可靠性（细节之美）

#### 挖掘方向

- **异常处理**：如何处理错误和边界情况？
- **调试机制**：有哪些调试/日志手段？
- **工程约束**：项目有哪些限制和已知问题？
- **可维护性**：代码的可读性和可扩展性如何？

#### Agent 自主发掘指令

```
请分析项目的工程实践：
1. 项目中有哪些错误处理和边界检查机制？
2. 调试和日志机制是如何设计的？
3. 项目有哪些已知的限制或 TODO 项？
4. 代码的可维护性如何？
```

## 第四章 组织与分类：三大素材分类箱

### 4.1 分类原则

根据素材的性质和用途，分为三类：

| 分类箱 | 定义 | PPT 用途 |
|--------|------|----------|
| **亮点箱** | 项目的创新点、技术亮点、优秀实现 | 核心技术、算法实现、优化对比 |
| **稳健箱** | 增强可靠性的设计、错误处理、测试 | 系统稳定性、可靠性设计 |
| **债务箱** | 已知问题、TODO、潜在改进点 | 局限性与展望 |

### 4.2 Agent 分类指令

```
请将发现的素材分类到以下三个箱子：

**亮点箱**：记录项目中的技术亮点
- 这个项目最值得展示的技术实现是什么？
- 有哪些体现创新或深度思考的设计？

**稳健箱**：记录增强可靠性的设计
- 项目中有哪些错误处理和容错机制？
- 有哪些保证系统稳定运行的设计？

**债务箱**：诚实记录待改进的地方
- 有哪些已知的限制或问题？
- 有哪些 TODO 或未来可以优化的方向？
```

## 第五章 数据与图表呈现规范

> **原则**：图表应具备"自解释性"，且必须包含**参照系**。数据呈现的核心是**工程验证**与**性能权衡**。

### 5.1 图表呈现的核心逻辑

Agent 在设计数据呈现时，应围绕以下逻辑展开：

| 呈现逻辑 | 说明 | Agent 应发掘的内容 |
|----------|------|-------------------|
| **基准对齐** | 与"金标准"对比 | 项目的参照基准是什么？误差是否在可接受范围？ |
| **敏感度分析** | 多参数的 Trade-off | 哪些参数会影响性能？它们的影响是什么？ |
| **复杂度验证** | 理论 vs 实测 | 理论预期与实际表现是否吻合？ |
| **资源效率** | 成本与收益的权衡 | 付出了什么代价？获得了什么收益？ |

### 5.2 深度呈现规范

#### 5.2.1 基准对齐（Golden Reference Comparison）

- **核心问题**：项目实现与"金标准"的差距是多少？
- **呈现方式**：误差图、对比曲线、差值分析
- **Agent 应发掘**：
  ```
  请识别项目的"金标准"参照：
  1. 有什么权威的参考实现可以对比？（官方库、理论公式、行业标准）
  2. 项目实现与参照的误差如何？
  3. 误差是否在可接受范围内？如何证明？
  ```

#### 5.2.2 敏感度分析（Trade-off Exploration）

- **核心问题**：参数变化如何影响系统行为？
- **呈现方式**：多图并列对比、热力图、等高线图
- **Agent 应发掘**：
  ```
  请识别项目中的关键可调参数：
  1. 哪些参数会显著影响系统性能？
  2. 这些参数之间是否存在 Trade-off 关系？
  3. 如何通过图表展示"没有最优参数，只有最适合场景的折中"？
  ```

#### 5.2.3 理论与实测对比

- **核心问题**：理论预测与实际表现是否一致？
- **呈现方式**：理论曲线 + 实测散点、拟合验证
- **Agent 应发掘**：
  ```
  请分析项目的理论基础：
  1. 项目是否有理论公式可以预测行为？
  2. 实测数据是否符合理论预期？
  3. 如果有偏差，原因是什么？
  ```

### 5.3 图表基本规范

| 规范项 | 要求 |
|--------|------|
| **图片格式** | 架构图用矢量图，截图用高清 PNG |
| **标注清晰** | 刻度、单位、关键点标注清晰 |
| **字体可读** | 确保远距离也能清晰阅读 |
| **参照系** | 图表必须包含对比基准 |
| **结论导向** | 图表应直接支撑你的结论 |

### 5.4 图表选择指引

Agent 应根据数据特性和呈现目的选择图表：

| 呈现目的 | 推荐图表 |
|----------|----------|
| 与金标准对比 | 误差曲线、差值图、叠加对比 |
| 参数敏感度 | 多图并列、热力图、3D曲面 |
| 趋势变化 | 折线图 |
| 分类对比 | 柱状图 |
| 多维评估 | 雷达图 |
| 理论验证 | 理论曲线 + 实测散点 |

## 第六章 叙事框架：如何讲好项目故事

> **原则**：避免平铺直叙，采用"**提出瓶颈 → 引入机制 → 验证改进**"的迭代式叙事。

### 6.1 工程演进叙事模式

好的项目叙事应该展现**工程演进过程**：

| 阶段 | 内容 | Agent 应发掘 |
|------|------|-------------|
| **初始方案** | 最基本的实现，建立基线 | 项目最初的实现是什么？基础性能如何？ |
| **发现瓶颈** | 识别问题和局限 | 初始方案有什么问题？瓶颈在哪里？ |
| **引入优化** | 针对性的改进措施 | 采用了什么技术来解决问题？为什么选择这种方案？ |
| **闭环验证** | 量化证明改进效果 | 改进后性能提升了多少？如何证明？ |

### 6.2 叙事生成的"黄金问题"

Agent 在构建叙事时，应回答以下问题：

```
请基于项目分析，回答以下黄金问题：

**Why this?** 为什么选择这种技术方案/架构？
- 与其他方案相比有什么优势？
- 做了什么权衡取舍？

**So what?** 这个改进/实现带来了什么具体效果？
- 可以量化的改进是什么？
- 如何通过图表直观展示效果？

**What's next?** 目前的方案还有什么局限？
- 有哪些已知的限制？
- 未来可以在哪些方向改进？
```

### 6.3 叙事弧线模板

```
**阶段一：问题定义**
- 我们要解决的问题是什么？
- 这个问题的挑战在哪里？

**阶段二：初始尝试**
- 最初采用了什么方案？
- 这个方案的表现如何？

**阶段三：发现瓶颈**
- 初始方案遇到了什么问题？
- 瓶颈的根因是什么？

**阶段四：改进策略**
- 采用了什么优化策略？
- 为什么这个策略能解决问题？

**阶段五：验证效果**
- 改进后的表现如何？
- 与初始方案相比提升了多少？

**阶段六：局限与展望**
- 目前方案还有什么限制？
- 未来有什么改进方向？
```

## 第七章 给 Agent 的综合行动指令

### 7.1 项目分析与素材挖掘（完整版）

```
Role: 你是一名系统软件工程师，擅长分析项目并挖掘汇报素材。

Task: 对以下项目进行分析和素材挖掘，为答辩或技术汇报准备素材。

**第一步：理解项目**
- 这个项目解决什么问题？
- 核心挑战是什么？
- 什么指标最能证明项目成功？
- 有什么"金标准"可以作为参照？

**第二步：发掘特性**
- 项目的核心模块和架构是什么？
- 有哪些技术亮点值得展示？
- 有哪些关键参数会影响性能？

**第三步：提取素材**
- 亮点箱：核心技术实现、优化手段、创新设计
- 稳健箱：错误处理、可靠性设计
- 债务箱：已知问题、改进方向

**第四步：量化建议**
- 如何与"金标准"进行对比验证？
- 哪些参数值得进行敏感度分析？
- 理论预测与实测是否需要对比？

**第五步：叙事组织**
- 项目的"工程演进"故事是什么？
- 从初始方案到优化方案经历了什么？
- 回答 Why this? So what? What's next?
```

|------|------|----------|

## 结语

**素材组织的核心不是填充模板，而是发掘项目的真正价值。**

Agent 应该：
1. **先理解**：深入理解项目的目标、挑战和特性
2. **找参照**：确定合适的"金标准"作为对比基准
3. **讲故事**：用"工程演进"的逻辑组织叙事
4. **证价值**：用量化数据和对比图表证明改进效果

每个项目都是独特的，好的素材挖掘应该让项目的独特价值脱颖而出。
